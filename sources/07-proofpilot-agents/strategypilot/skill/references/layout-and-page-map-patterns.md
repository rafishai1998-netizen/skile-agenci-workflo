# StrategyPilot Layout and Page Map Patterns

Use this reference when you want the strategy to feel more like a clean page blueprint and less like a loose memo.

## Core Rule

Do not turn the strategy into a proposal.
That means:
- no add-ons section
- no pricing section for deliverables
- no package breakdowns
- no sales-offer framing

But DO borrow the strongest layout ideas from great strategy and proposal docs:
- clear summary table near the front
- sharp strategic focus box
- clean page-map tables
- obvious hierarchy between hubs, subpages, and targeting

## Best Front-of-Doc Layout Pattern

After the cover page, include a simple summary table with fields like:
- Prepared For
- Domain
- Market or Region
- Main Focus
- Current Footprint
- Recommended Direction
- Date

This helps the reader orient quickly.

## Strategic Focus Box

Early in the doc, use a small strategic focus box to show what the strategy is really about.
Examples:
- high-ticket services
- core local markets
- best geographic targets
- strongest service clusters
- commercial growth focus

This should feel like a strategic lens, not a sales pitch.

## Page Map Presentation Rules

When recommending pages, do not just use bullets.
Use clean tables and grouped sections.

Strongest visual pattern to borrow:
- bold section title
- one-line strategic statement under it
- grouped roadmap table beneath it
- rows organized by service family or priority tier
- page names on the left, implementation detail in the middle, strategic value on the right

For strategy docs, do not rely on a status column like INCLUDED or SEO ADD-ON.
That is proposal language.
Instead, the far-right column should usually be one of these:
- main keyword
- estimated local monthly search volume
- why it matters

## Recommended table types

### 1. Core site structure table
Columns:
- Page Type
- Purpose
- Targeting Focus
- Priority

Good for:
- homepage
- service hubs
- location hubs
- trust pages
- blog or resource hubs

### 2. Service priority matrix
Columns:
- Service Category
- Why It Matters
- Main Keyword or Search Theme
- Priority

Use this when you need to show which service families deserve the most attention first.

### 3. Service roadmap table
Columns:
- Page
- URL
- Main Keyword or Search Volume

Use this for hub pages and child service pages.
This is one of the cleanest formats for strategy docs because it feels structured and implementation-ready.

### 4. Location page table
Columns:
- Location Page
- Main Service Focus
- Main Keyword or Search Volume
- Priority

### 5. Service plus city matrix
Columns:
- Service
- City
- Page to Build
- Main Keyword or Search Volume

This is one of the strongest ways to make the strategy actionable.

### 6. Content pillar table
Columns:
- Pillar Topic
- Supporting Spoke Topics
- Main Money Page Supported
- Why This Pillar Matters

### 7. Internal linking map
Columns:
- Hub Page
- Supporting Pages
- Links To
- Role In SEO

## Local Service Website Structure Pattern

For local service businesses, the page map should usually be presented in this order:
1. homepage
2. main service hub or residential/commercial hubs
3. core service pages
4. location pages
5. service plus city pages
6. proof pages
7. blog or FAQ pillars

This matches how business owners think about their site.

Presentation pattern that works especially well:
- start with a service priority matrix
- then show one service family at a time
- use a hub row first
- place child pages directly underneath the hub row
- group each service family under a heading like "Scorpion Services (Priority 1)"
- if the table continues across pages, keep the same column widths and repeat the table header on the next page

Matthew preference, make hierarchy painfully obvious:
- do not label something only as a "service cluster" if the reader still has to guess what the main page is
- explicitly show the main category page first, then the subpages underneath it
- for service families, prefer columns like:
  - main category page
  - subpages underneath it
  - why this hub matters
  - estimated local monthly search volume or keyword demand
- for service-plus-city plans, do not scatter isolated service/city rows if grouping by service would make the hierarchy clearer
- instead, show the main service first, then list the city pages underneath that service
- if a matrix is hard to scan, collapse it into grouped rows by parent page or parent service
- the reader should instantly understand what the parent page is and what children sit under it without needing explanatory prose

## How to Explain Hubs and Subpages

Keep it simple.
Use language like:
- This is your main service page.
- These are the sub-service pages under it.
- This is your city hub.
- These are the location pages under it.
- These blog posts support this core page.

Avoid overcomplicated SEO wording if a simpler label works.

## Good Section Pattern for Strategy Docs

A clean strategy often works best in this order:
1. Executive Summary
2. What Is Working
3. What Is Weak
4. Strategic Focus
5. Core Site Structure
6. Service Page Plan
7. Location Page Plan
8. Service Plus City Plan
9. Content Pillars and Spokes
10. Internal Linking and Architecture
11. Revenue and ROI Model
12. Roadmap

Not every doc needs every section, but this is a strong default pattern.

## Rich Visual Layout Patterns

When the document needs to feel more designed, not just well written, these patterns work well:
- visual sitemap pages with stacked hierarchy boxes
- primary navigation mockups for top-level site structure
- service priority matrix pages
- grouped service-family roadmap pages
- repeated page markers and brand lines for continuity across longer docs
- continuation tables that preserve the same column widths and styling across pages

Use these sparingly and only when they improve clarity.
The goal is to make the strategy easier to understand, not to decorate it.

### Premium visual rules that translate well into Google Docs or python-docx

If Matthew wants the strategy to feel closer to a deck or premium proposal layout, use these presentation rules:
- prefer symmetrical table-based layouts over loose floating elements
- for hierarchy pages, stack 2-column tables vertically instead of trying to fake complex diagrams
- use a dark / bright / light tier system to show parent, child, and detail levels
- use bold condensed all-caps headings for section titles and hierarchy labels
- use clean sans-serif body text for URLs, captions, and explanatory copy
- use thin light-gray borders, not heavy spreadsheet borders
- keep generous whitespace between rows, arrows, and tables so the page does not feel cramped
- when showing search volume, make the volume easy to scan, like a dedicated column or clear inline estimate, instead of hiding impact in vague labels
- repeat footer brand lines and page markers to make longer docs feel more intentional
- keep continuation tables identical in column widths and styling so the doc reads like a designed system

### Rich layout still has to be grounded

A richer layout should not just make the document prettier. It should make the diagnosis easier to trust.
That means:
- use the richer pages to show the real hierarchy, not decorative shapes
- tie visual sitemap pages directly to the actual build order
- surface obvious template residue or unfinished sections clearly in the diagnosis pages
- make sure premium formatting does not hide weak reasoning or generic recommendations
- if the site has clear quality problems like placeholder content, cross-market template leftovers, or city-page duplication, show those plainly before moving into the prettier roadmap pages

### Specific layout pattern Matthew wants for future StrategyPilot docs

When a strategy section is about service families, service-plus-city plans, or site architecture, upgrade the layout beyond a generic matrix:
- use grouped family blocks that start with the hub page first
- make the hub page visually explicit with labels like `(Hub)`
- show child pages underneath with arrow-style rows like `→ Subpage Name`
- prefer practical columns like:
  - Page
  - URL
  - Priority or status
- after the grouped family tables, add a short build-order table so the execution sequence is obvious

For local page architecture:
- when helpful, use a table-based pseudo-flowchart instead of only prose
- show the main hub first, then arrows, then mid-level hubs, then city groups underneath
- a strong pattern is:
  - one main hub card or row
  - arrow down
  - 2-4 service or location hub cards
  - arrow down
  - lower-level gray boxes or grouped rows for cities, submarkets, or support pages
- then follow the visual hierarchy with a practical URL table

The rule:
- the reader should be able to tell what the top page is, what sits under it, and what gets built next in under 10 seconds

## What to Borrow, What Not to Borrow

### Borrow
- clear front-of-doc summary table
- obvious structure
- page counts or page-family framing when useful
- grouped page recommendations
- page targeting tables
- content pillar tables
- clear hub and spoke presentation
- service priority matrices
- grouped service-family roadmap tables
- page, URL, and keyword/search-volume formatting
- continuation-table consistency across pages

### Do not borrow
- proposal pricing language
- included deliverables pricing grids
- upsell sections
- add-on bundles
- status labels that feel like package sales language when search volume or keyword targeting would be more strategic
- anything that makes the strategy feel like a quote instead of a blueprint

## Final Check

Before finalizing, ask:
- Does the strategy feel easier to scan?
- Can the client quickly see what pages to build?
- Are hubs, subpages, and targeting clearly grouped?
- Did I improve the presentation without turning it into a proposal?
