# GBPPilot No-Miss Audit Checklist

Use this before finalizing any GBP audit or strategy document.

## Non-negotiable audit questions

Every GBP audit should explicitly answer these, even if the answer is "not visible" or "unverified":

1. **Posts**
- How many owner posts are visibly present in the last 30 to 60 days?
- If none are visible, does the document say that plainly?
- If the surface is blocked or limited, is the backend state labeled unverified instead of guessed?

2. **Products**
- Are any products visible publicly?
- If yes, how many are visible and are they strong hero offers or weak filler?
- If none are visible, does the document frame that as a missed frontend and backend opportunity?

3. **Services**
- Is a public services list visible?
- If yes, were the exact visible labels captured first?
- Were duplicates or messy labels collapsed into clean canonical service clusters?
- Does the report include a service-to-website mirroring matrix?

4. **Areas served**
- Is an areas-served list visible publicly?
- Were the exact cities, neighborhoods, or zip codes captured?
- Does the report say whether obvious priority cities are missing or whether the list looks padded?

5. **Geo relevance**
- Is there direct geo-grid evidence such as Local Falcon for at least one primary non-branded keyword?
- If not, does the report avoid making hard heat-map claims and instead frame this as a recommended next step?

6. **Citations and NAP**
- Does the audit include a quick branded citation check when NAP trust looks weak or inconsistent?
- Are citation findings tied to evidence rather than generic SEO advice?

7. **Mirroring strategy**
- Does the document show table-first mapping from GBP services to website pages?
- Are CREATE pages clearly marked?
- Are duplicate or overlapping GBP labels consolidated into a sane site architecture?

8. **Location-page discovery**
- Did the audit verify location pages with more than one method when the site structure looked unclear?
- Did it avoid assuming there were no location pages just because one sitemap summary looked thin?

## Public-view vs backend truth rule

Never treat the public Google Maps view as a full backend export.

Use these labels consistently:
- **Visible**: confirmed in the public profile
- **Likely**: inferred from supporting evidence
- **Recommended**: the correct strategic setup
- **Unverified**: cannot be proven from the available public view

## Google anti-bot fallback rule

If a Google share link triggers a block or limited view:
1. try the direct Google Maps place URL
2. pull Maps URLs from the business website if available
3. use `web_extract` on the Maps place URL to capture public panel text
4. use browser tools on the place URL instead of the generic share link when possible
5. continue with website, competitor, and strategy analysis instead of stopping

## Final deliverable rule

The final doc should not only say what is wrong.
It should show:
- what is visible now
- what the correct strategy should be
- what the website must mirror one to one
- what gets fixed first
